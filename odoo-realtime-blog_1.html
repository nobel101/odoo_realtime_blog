The Browser That Wouldn't Stop Asking

How a fundamental limitation of HTTP forced the web to reinvent itself — and how Odoo quietly rebuilt its entire real-time layer around the solution.

12 min read  ·  Long polling · WebSockets · im_bus · Odoo 16

In 2004, if you wanted to know whether a new email had arrived in your Gmail inbox, your browser had to ask. It couldn't be told. It couldn't listen. It could only ask — and so it did, every few seconds, in a loop, forever.

This wasn't a bug. It was the entire design of the web.

HTTP, the protocol that carries every web page you've ever loaded, was built around a simple idea borrowed from postal mail: you send a request, you receive a response, the connection closes. Clean, stateless, scalable. Perfect for fetching documents. Completely wrong for a world where software needs to push events to users in real time.

Twenty years later, Odoo — one of the world's most widely deployed ERP systems — quietly completed a multi-version journey through every stage of that evolution. Understanding why it made those changes, and how it implemented them, means understanding one of the more interesting chapters in the history of web networking.

· · ·

The Stateless Contract

To understand the problem, you have to go back to the protocol itself.

HTTP was designed by Tim Berners-Lee in 1989 as a way to link and serve documents across a network. The model was elegantly simple: a client sends a request, a server sends a response. When the response is delivered, the connection is finished. The server has no memory of the client. The client has no persistent relationship with the server. Every interaction starts from zero.

This statelessness was a feature, not a limitation. It made servers massively scalable — you could handle thousands of clients without keeping any state for any of them. It made caching trivial. It made the protocol fault-tolerant; if a request failed, you just sent it again.

The web was designed to serve documents. Nobody told it that one day it would need to run a live chat window inside an inventory management system.

For most of the 1990s, this was fine. Websites were documents. You loaded a page, you read it, you clicked a link, you loaded another page. The request-response cycle was the entire user experience.

Then came web applications. And with them, a question the protocol had no answer for: what if the server needs to tell the client something that the client didn't ask about?

· · ·

The Polling Era: When the Browser Became a Pest

The first solution was brute force. If the browser can only receive information by asking, make it ask constantly.

This technique — called short polling — was the standard approach for any application that needed live updates through the early 2000s. A JavaScript timer would fire every few seconds, send an HTTP request to the server, and check for new data. If there was something new, render it. If not, throw away the response and wait for the next timer tick.

Short polling in practice — circa 2003
setInterval(function() {
  fetch('/check-for-new-messages')
    .then(r => r.json())
    .then(data => { if (data.messages.length) render(data); });
}, 3000);

This fires 20 requests per minute, per browser tab, per user.
With 500 concurrent users: 10,000 HTTP requests per minute — most returning an empty response.

The waste was staggering. For every useful response containing actual new data, dozens of empty responses were discarded. Each request carried a full HTTP overhead: headers, TCP handshake costs, server-side processing to check the database. At any meaningful user count, the server spent most of its CPU answering the question "anything new?" with "no."

But for a long time, this was the only option. The web had no other model.

· · ·

The Comet Trick: Holding the Line Open

Sometime around 2006, developers started noticing something interesting about HTTP. While the protocol specified that connections should close after a response, nothing in the TCP layer required them to. A server could technically start sending a response and then just... not finish it. Hold the connection open. Send more data when it became available.

Alex Russell, a developer at Dojo Toolkit, wrote a blog post in 2006 naming this pattern Comet — a riff on Ajax, as in another household cleaning product. The name stuck.

The elegant variant of this approach was called long polling. Instead of the browser asking every three seconds and getting a response immediately, the browser would ask once — and the server would hold the request open, doing nothing, until either new data arrived or a timeout occurred. When data arrived, the server would flush the response. The browser would immediately open a new long-poll request. And the cycle would continue.

The long-poll conversation
Browser → Server: "Anything new?"
Server: (holds connection, waits... waits... waits...)
30 seconds later, a new chat message arrives:
Server → Browser: "Yes, here it is." (connection closes)
Browser → Server: "Anything new?" (immediately re-opens)

Result: Near-real-time delivery. One active connection instead of a polling storm.

This was genuinely clever. The number of requests dropped dramatically. Latency — the gap between an event occurring on the server and the browser learning about it — dropped from several seconds to near-zero. All of this without changing the HTTP protocol at all.

But long polling was a hack. It worked against the design of the protocol rather than with it. Servers had to keep a thread or process alive for each waiting connection. Load balancers would sometimes kill connections they thought had stalled. Proxies would buffer responses and refuse to forward them until the connection closed. The whole arrangement was fragile in ways that were hard to predict and harder to debug.

· · ·

The Year the Protocol Finally Changed

In 2011, the IETF published RFC 6455: The WebSocket Protocol.

It was a proper, standardized solution to the problem that Comet had been patching over for five years. WebSocket defined a handshake mechanism — built on top of a regular HTTP upgrade request — that converted an existing TCP connection from the HTTP model into a persistent, full-duplex channel. Once the handshake was complete, either side could send a message at any time, in either direction, with minimal overhead.

1989
HTTP invented at CERN. Request → Response → Close. The stateless web is born.
1995
JavaScript arrives. The browser gains logic. Web pages start needing live data.
2003
Short polling becomes standard practice. Browsers hammer servers every few seconds.
2006
Alex Russell names Comet / long polling. The server holds connections open — a clever abuse of HTTP.
2009
Google Wave demonstrates full real-time collaboration. WebSocket spec begins serious development.
2011
RFC 6455 published. WebSocket is a real protocol. Browsers start shipping it by default.
2013
All major browsers fully support WebSocket. The hack era is over — for those who adopt it.
2022
Odoo 16 ships with WebSocket as the default transport. Long polling becomes the fallback.

The key innovation in WebSocket wasn't the persistent connection — long polling had that. It was the framing protocol. Where long polling sent complete HTTP responses and had to re-establish the connection each time, WebSocket sent lightweight frames. A single frame could be as small as 2 bytes of overhead. The TCP connection stayed open indefinitely, maintained by periodic ping/pong heartbeat frames. The server could push a message to the browser at any moment — no waiting, no request needed.

· · ·

Where Odoo Was During All of This

Odoo — known as OpenERP until 2014 — was not a web application in its early years. It was a thick-client GTK desktop application that happened to have a web interface bolted on. Real-time communication was not a design concern when your interface was a native window that could hold a persistent connection to the server however it liked.

The shift happened gradually as Odoo committed fully to the browser as its primary interface. And with that shift came the problem every web developer eventually faces: how do you tell the browser that something happened?

For years, the answer was the one everyone else was using. Poll the server. The Odoo client would ping endpoints on a timer, checking for new messages, pending activities, and system notifications. It worked well enough at small scale — a few dozen users, a quiet server, a forgiving network.

But Odoo was being deployed at larger and larger scales. And at scale, polling has a physics problem: the number of requests grows linearly with your user count. A deployment with 500 concurrent users generates as many empty polling requests per minute as a deployment with 50 users generates useful ones.

· · ·

The Architecture of im_bus: Designing for Decoupling

Before Odoo could adopt WebSockets, it needed something more fundamental: a proper internal event system. The transport layer — whether long polling or WebSocket — would be meaningless without a coherent way for any part of the application to say "something happened," and for any connected client to receive that message.

The answer was the im_bus module, and its central model: bus.bus.

The design philosophy was straightforward: make the bus a database table, not an in-memory queue. This was a deliberate choice that traded raw performance for reliability and architectural simplicity. Any Odoo server process — a web worker, a cron job, a background task — can publish an event by inserting a row into bus_bus. No inter-process communication required. No shared memory. No message broker to manage. Just a database write.

◈

Using PostgreSQL as the message queue was a pragmatic choice. Odoo already required PostgreSQL. Every server process already had a database connection. The consistency guarantees of a transactional database meant that events could never be silently lost — if the write committed, the event existed.

But a database table alone doesn't push anything. Something still needs to notice when new rows appear and forward them to connected clients. This is where Odoo used one of PostgreSQL's most underused features: LISTEN and NOTIFY.

PostgreSQL LISTEN/NOTIFY — the wakeup mechanism SQL
-- The bus worker runs this once on startup:
LISTEN im_bus_notification;

-- When any process writes to bus_bus, it triggers:
NOTIFY im_bus_notification;

-- The LISTEN connection wakes up immediately.
-- No polling. No sleep loop. Pure event-driven wakeup.

The effect is elegant. The bus worker process maintains a dedicated PostgreSQL connection that does nothing except wait for a NOTIFY signal. When any part of the application writes a message to bus_bus, it fires a NOTIFY. The sleeping worker wakes up, reads the new rows, and pushes them to the appropriate connected clients. The worker then goes back to sleep. Zero wasted CPU between events.

Long Polling on Top of im_bus

With im_bus in place, the long-polling implementation became straightforward. The browser would send a request to /web/bus/poll, passing the ID of the last message it had seen. The server would check whether any newer messages existed for the client's subscribed channels. If not, it would hold the HTTP connection open — waiting on the PostgreSQL LISTEN connection — until either a new message arrived or a timeout (typically 50 seconds) expired.

The long-poll request lifecycle in Odoo
1. Browser → POST /web/bus/poll with {"last": 48291, "channels": ["mail.channel/7", "partner#42"]}
2. Server checks bus_bus for rows with id > 48291 matching those channels → nothing found
3. Server suspends the HTTP response; LISTEN connection sleeps
4. Another user sends a chat message → bus_bus row inserted → NOTIFY fires
5. Server wakes, reads new row, flushes HTTP response with the message
6. Browser receives message, immediately re-issues POST /web/bus/poll with {"last": 48292, ...}

This was a substantial improvement over dumb polling. But it still had the fundamental problem of long polling: every connected client occupied a server thread waiting on an HTTP connection. In a standard synchronous Python web server, threads are expensive. An Odoo deployment with 200 simultaneous users meant 200 threads doing nothing but waiting.

Odoo worked around this by using gevent — a Python library that uses cooperative multitasking to handle thousands of concurrent connections on a small number of OS threads. A gevent-based Odoo worker could manage hundreds of waiting long-poll connections efficiently. But this required a separate server process, careful configuration, and still had ceiling it couldn't break through cleanly.

Odoo 16: The WebSocket Migration

By the time Odoo 16 was in development, WebSockets had been stable and universally supported for nearly a decade. The question was no longer whether to use them — it was how to integrate them cleanly into an existing architecture without breaking the applications already built on top of im_bus.

The answer was to treat the WebSocket as a transport layer, not as a replacement for the bus. The im_bus channel system stayed exactly the same. Module developers continued to publish events the same way. The only thing that changed was the pipe connecting the browser to the bus worker.

Before Odoo 16 — Long Poll
  • New HTTP connection every ~50 seconds
  • Full request headers on each reconnect
  • Server thread occupied while waiting
  • Proxy timeouts can kill idle connections
  • Gevent required for scale
Odoo 16+ — WebSocket
  • One connection per tab, held indefinitely
  • 2-byte frame overhead per message
  • Async worker handles thousands of connections
  • Ping/pong heartbeat detects dead connections
  • Falls back to long poll if WS fails

The WebSocket handshake in Odoo begins as a regular HTTP request. The browser sends an Upgrade: websocket header. The server responds with 101 Switching Protocols. From that moment, the TCP connection no longer speaks HTTP — it speaks the WebSocket frame protocol. The Odoo session, authentication, and channel subscriptions are all handled in the first few frames after the upgrade.

WebSocket handshake — what actually travels on the wire HTTP/1.1
Browser sends:
GET /websocket HTTP/1.1
Host: odoo.company.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

Server responds:
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

From this point, the connection is no longer HTTP.
Odoo sends a subscribe frame; the browser acknowledges.
The server can now push messages at any time.

The subscribe frame is where Odoo's bus logic connects to the WebSocket transport. The JavaScript bus.service sends a JSON frame listing all the channels the current user cares about — their partner channel, active mail threads, notification channels. The server registers these subscriptions in memory, associated with that WebSocket connection. When a message arrives in bus_bus for any of those channels, the WebSocket worker pushes a frame immediately.

The Kubernetes Wrinkle

Anyone who has deployed Odoo 16+ behind an NGINX ingress in Kubernetes has likely hit this: WebSockets silently fail, and Odoo reverts to long polling with no explanation.

The cause is almost always the ingress stripping the Upgrade and Connection headers from the HTTP request before it reaches the Odoo pod. These headers are hop-by-hop headers — they're meant to be consumed and re-emitted by each proxy in the chain, not forwarded blindly. Many default NGINX configurations strip them entirely.

Without these headers, the server never sees a WebSocket upgrade request. It sees a regular HTTP request to /websocket, returns a 400 or 404, and the Odoo client quietly falls back to /web/bus/poll. Everything still works. You just never get the benefits of WebSocket, and you'll wonder why your server has hundreds of long-poll connections open at all times.

nginx ingress — the required annotations YAML
annotations:
  nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
  nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"
  nginx.ingress.kubernetes.io/configuration-snippet: |
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";

The timeout values matter too. A default proxy timeout of 60 seconds will kill any WebSocket connection that hasn't had traffic in a minute, even if both sides are alive and the Odoo heartbeat is configured properly. Setting read and send timeouts to 3600 seconds — one hour — gives the WebSocket connection room to breathe between low-traffic periods.

What This Means for Module Developers

The most interesting thing about this entire architecture is what it hides. A developer writing an Odoo module in 2024 doesn't need to know any of this. They call one method:

Publishing an event from any Odoo module Python
# In any Odoo model method:
self.env['bus.bus']._sendmany([
    (f'mail.channel/{self.id}', 'new_message', {
        'id': message.id,
        'body': message.body,
        'author': message.author_id.name,
    })
])
# That's it. Whether the client is using WebSocket or
# long poll, it will receive this within milliseconds.

The transport abstraction is complete. The module developer doesn't know — and doesn't need to know — whether the client receiving this message is connected via WebSocket or long poll. The bus handles it. The history of two decades of web networking evolution is invisible behind a single method call.

◈

On the JavaScript side, subscribing is equally simple. Any Owl component or service can call this.env.services.bus_service.subscribe('new_message', handler) on a channel. The bus service manages the WebSocket connection, handles reconnection with exponential back-off, and delivers events to handlers — abstracting the entire transport layer away from the application code.

The Full Picture

Here is the complete architecture — browser on the left, server on the right, the bus sitting in the middle of everything.

What began as a browser limitation — the stateless contract of HTTP — forced an entire industry to get creative. Short polling gave way to long polling. Long polling gave way to WebSocket. Each transition reduced waste, improved latency, and lowered server load. The pattern repeated itself in every major web framework, in every language, across two decades.

Odoo's journey through this arc was slower than some and more careful than others. The decision to keep the bus as a database table — rather than adopting Redis or RabbitMQ as a message broker — was unusual, even eccentric. But it meant zero additional infrastructure dependencies. It meant events were transactional. It meant any DBA could inspect the message queue with a simple SQL query.

The decision to build WebSocket as a transport layer on top of the existing bus, rather than replacing the bus entirely, meant that thousands of modules written for the long-poll era continued to work without modification.

These are not the most glamorous architectural decisions. They won't appear in conference keynotes about distributed systems. But they are the kind of decisions that make a system quietly reliable for the hundreds of thousands of companies that run Odoo in production — and never think about any of this for a single moment.

Which is, of course, exactly the point.
